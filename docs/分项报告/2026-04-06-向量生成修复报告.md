## 2. 根因分析

### 2.1 错误定位

通过 API 调用测试发现：

| 批量大小 | HTTP 响应 |
|----------|----------|
| 5 条文本 | 200 OK |
| 10 条文本 | 200 OK |
| 20 条文本 | 200 OK |
| 30 条文本 | 200 OK |
| **35 条文本** | **413 Request Entity Too Large** |

**SiliconFlow 单次请求文本量上限 < 30 条。**

### 2.2 原脚本逻辑缺陷

```python
# 原脚本（gen_vector_sf.py）
BATCH_SIZE = 50   # ❌ 超过上限 30

# 当 HTTP 413 发生时：
except Exception as e:
    print(f"[重试 {attempt+1}/{MAX_RETRIES}] {e}")
    time.sleep(2)
# 第3次重试失败后... 循环重新进入 while True，但 remaining > 0
# → 再次 SELECT LIMIT 1 → 拿到同一批数据 → 再次 413 → 无限循环卡住
```

核心问题：**HTTP 413 后脚本卡在第一批数据上反复重试，无法推进**。

---

## 3. 解决方案

### 3.1 修复策略

新建 `gen_vector_fast.py`，核心改动：

| 项目 | 原脚本 | 新脚本 |
|------|--------|--------|
| 批量大小 | 50 | **20** |
| 并发 | 1（逐条）| **3 线程** |
| API 调用 | 单条 | **批量** |
| 错误处理 | 重试 2 次后卡住 | 重试 3 次后回滚跳过 |

### 3.2 关键代码

```python
BATCH_SIZE = 20       # SiliconFlow 实测上限 30 以内
MAX_WORKERS = 3       # 并发线程

def embed_batch(texts):
    """批量调用，返回 embeddings 列表"""
    payload = {'model': 'BAAI/bge-large-zh-v1.5', 'input': texts}
    req = urllib.request.Request(API_URL, data=json.dumps(payload).encode(), ...)
    with urllib.request.urlopen(req, timeout=60) as r:
        result = json.loads(r.read())
        embeddings = result['data']
        embeddings.sort(key=lambda x: x['index'])  # 保证顺序
        return [e['embedding'] for e in embeddings]
```

## 5. 经验总结

| 教训 | 说明 |
|------|------|
| **先测 API 上限再写脚本** | 未实测批量上限就按 50 条写，导致 413 |
| **批量比逐条快 500+ 倍** | 20 条批量 vs 逐条，API 往返开销大幅降低 |
| **并发进一步提速 3 倍** | 3 线程并发，API 吞吐提升明显 |
| **错误后要回滚，不能卡住** | 原脚本错误处理不完善，陷入死循环 |
| **环境 Python 版本影响大** | Python 3.9 无法连接 Postgres.app（trust 认证），Python 3.11 正常 |
